fake 테스트는

테스트 하려는 프로덱션 코드가 의존하고 있는 의존성들을 입출력만 같도록 fake 의존성으로 구현한다. (실제 DB 접근등 X)

fake 의존성을 프로덕션 코드에 주입하고 실행시킨다.

프로덕션 코드 내부 함수 호출흐름, 내부구현엔 영향을 잘 받지 않고, 결과값이 같으면 
	성공하는 블랙박스 테스트 성향을 띈다. (정규화, 함수 호출순서 등이 상관 없이 결과만 같으면 됨.)



## 안깨지는 좋은 코드를 만들기 위해선 어떤게 좋을까?

=> 블랙박스 테스트로 짜야지 안깨지는 좋은 테스트 코드를 짤 수 있다.(대부분 실무에서 이렇게 씀.)
블랙박스 테스트 == 내부 구현을 상관하지 않고, 결과로 테스트를 판단함.

## mock
mock 은 기본적으로 Behavior Verification(행동검증) 방식이다. 
-> 메서드의 호출/순서 등을 테스트 (기본적으로 화이트 박스 테스트 성격이 강함.)

블랙박스 테스트 형식으로 짜려면 verify 필수적인 메서드 이외에 사용하면 안되고 , every 스텁은 createMemo( any:Any ) 같이 느슨하게 짜야하고, 정규화에 맞춰서 every 를 여러개 만들어 놓는다는등의 의식적인 노력을 해야함.


## Fake
Fake 는 기본적으로 State verification(상태검증) 방식이다. -> 프로덕션 코드의 결과 상태를 테스트



fake 는 내부 구현이 아닌, 최종 결과(상태) 에 따라
